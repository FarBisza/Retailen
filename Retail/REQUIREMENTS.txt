3) Замечания / потенциальные проблемы (важные)
3.1 Дублирование и расхождения модулей (orderApi vs paymentApi)

У тебя есть и orderApi.ts, и paymentApi.ts, и они частично дублируют одно и то же:

    checkout есть в обоих (и даже возвращаемые типы разные: total vs totalAmount).
    payOrder есть в обоих.
    getInvoice есть в orderApi.ts и отдельно в adminApi.ts (и модели разные).

Это создаёт риск, что в UI разные страницы будут использовать разные клиенты и “поедут” типы/контракты.

Рекомендация: оставить один источник истины:

    либо “orderApi = клиентские заказы + оплата + invoice”,
    либо “paymentApi = оплата/инвойсы/шипменты”, но тогда orderApi без pay/checkout.
    И унифицировать модели ответа.

3.2 Непоследовательность endpoints по логистике/отгрузкам

В logisticsApi.ts:

    markShipped → POST /logistics/shipments/{id}/ship
    markDelivered → POST /logistics/shipments/{id}/deliver

А в paymentApi.ts:

    POST /logistics/shipments/{id}/mark-shipped
    POST /logistics/shipments/{id}/mark-delivered

Это похоже на “легаси/старые маршруты” или несостыковку. Лучше привести к одному варианту (и удалить второй), иначе легко словить 404/405 в одной из частей UI.
3.3 Неодинаковая обработка ошибок

Где-то ты:

    делаешь throw new Error('Failed...'),
    где-то читаешь res.text() и включаешь в сообщение,
    где-то возвращаешь []/null без исключения (например returns/category).

С точки зрения поддержки лучше сделать единый подход:

    один helper apiFetch<T>(url, options): Promise<T> который:
        добавляет headers,
        парсит ошибки (как твой parseErrorResponse),
        и стандартизирует поведение (либо всегда throw, либо всегда Result-типы).

3.4 reviewApi.ts обходит общий механизм headers

reviewApi.ts вручную берёт токен и делает fetch на /api/product/.../review, не используя getHeaders():

    потеряется X-Session-ID (может быть не важно для review),
    будет другой стиль обработки ошибок,
    риск расхождений при будущих изменениях auth.

Лучше перевести на getHeaders() для единообразия.
3.5 Типы местами “плывут”

    orderApi.checkout(address?: any) — any лучше заменить на интерфейс ShippingAddress/CheckoutAddress DTO.
    В types.ts есть Order/OrderItem, но похожие типы определены ещё и в orderApi.ts. Это источник путаницы (одно и то же имя — разное содержимое).

Рекомендация: вынести DTO в один файл (например dto/*.ts) и переиспользовать.
4) Безопасность (коротко, но важно)

    JWT хранится в sessionStorage. Это стандартно для прототипа, но уязвимее к XSS, чем HttpOnly cookie.
    Ты упоминал Refresh Token в HttpOnly cookie в тексте работы — ок. Тогда важно, чтобы стратегия была консистентной:
        если сессия реально держится на refresh-cookie, надо понимать, отправляются ли cookies всегда (при API_URL='/api' на том же origin — да, по умолчанию credentials: 'same-origin').
        но у тебя credentials: 'include' указан только в logisticsApi — это выглядит несимметрично. Если когда-то API станет cross-origin, часть запросов начнёт “терять” cookie.

### 3.1 Потенциальный баг: `openCreatePoWithProduct` затирает форму PO не полностью
`createPoData` типизирован как объект с полями:

А в `openCreatePoWithProduct` ты делаешь:
```ts

  supplierId: suppliers[0]?.id || 0,
  items: [...]
```

Тут **теряются** `warehouseId`, `expectedDate`, `comment`. В зависимости от TS-настроек это либо уже должно ругаться, либо приведёт к “поломанному” состоянию формы/создания PO.

Исправление: заполнять все поля (или делать merge с prev):
```ts
setCreatePoData(prev => ({
  ...prev,
  supplierId: suppliers[0]?.id || prev.supplierId,
  items: [...]
}));
```

Из-за этого легко пропустить ошибки типа “role может быть null”, “isActive отсутствует” и т.п. Лучше:

```ts
const [users, setUsers] = useState<User[]>([]);
```

## 2) Важные несостыковки/риски (по этому куску)

### 2.1 Логика статусов PO ↔ разрешение GR
Сейчас **GR разрешён при `po.statusName === 'Confirmed'`**:

```tsx
po.statusName === 'Confirmed' ? openGrForPo(po) : ...
```

Но по твоему же описанию процесса (и типовой логике) приёмка PZ должна быть доступна **после доставки/прибытия**, то есть минимум на статусе типа `InDelivery/Arrived/Delivered`.

Если на backend приёмка реально разрешена только после “прибытия”, то UI будет вводить в заблуждение и получать 409/400. Если backend разрешает GR на Confirmed — тогда процесс упрощён (это можно честно описать в тексте как допущение прототипа).

Рекомендация: привязать доступность GR к более корректному статусу (или хотя бы к статусId), и/или добавить кнопку “Mark Arrived” (или аналогичный шаг), чтобы PO переходил в состояние “можно принимать”.

### 2.2 “Ledger Impact (Preview)” считает увеличение неправильно
В preview:

```ts
Inventory Increase = sum(quantityReceived)
```

Но в реальности в склад обычно должно идти **net accepted**, т.е. `quantityReceived - quantityDamaged`. Иначе UI обещает “+100 units”, а на склад реально попадёт 95.

То же самое с “Inbound Value”: ты считаешь по `quantityOrdered`, а логичнее по факту (`quantityReceived` и, возможно, минус damaged).

### 2.3 Analytics: есть потенциальный баг с `p.isActive`
На вкладке analytics:

```ts
const activeProducts = products.filter(p => p.isActive).length;
```

Но в твоём `types.ts` интерфейсе `Product` нет поля `isActive` (и в `mapProduct` из `productApi` оно тоже не маппится). Это значит:
- либо TS это пропускает из-за слабой типизации где-то (тогда риск багов),
- либо `activeProducts` всегда будет 0.

Рекомендация: либо добавить `isActive?: boolean` в `Product` и реально передавать с backend, либо использовать существующее (`inStock`, `stockLevel`, etc).

### 2.4 Термин “Total Revenue” в analytics не соответствует вычислению
```ts
totalRevenue = sum(price * stockLevel)
```
Это не revenue, а скорее **Inventory Value (оценка стоимости складских остатков по текущей цене)**. Лучше переименовать переменную/лейбл, чтобы не было логической ошибки в BI-разделе.

### 2.5 Returns: “Approve & Refund” делает два запроса подряд
```ts
await handleReturnStatusUpdate(id, 2, amount);
await handleReturnStatusUpdate(id, 4, amount);
```
Риски:
- гонки/неконсистентность, если второй запрос упадёт
- лишние round-trip
- возможны бизнес-ограничения на backend (например нельзя сразу в 4 без проверок)

Лучше иметь backend-операцию “approveAndRefund” или транзакцию на сервере, либо хотя бы один endpoint “set status” с проверенными переходами.

### 2.6 Create PO modal не даёт выбрать warehouseId
В стейте `createPoData` есть `warehouseId`, но в модалке выбора склада нет. Сейчас warehouseId фактически всегда остаётся дефолтным (или может стать undefined после других действий, как я отмечал раньше).

- В UI исключаются уже назначенные атрибуты (фильтр по `attributeId`), что снижает риск дублей.

Это хороший практический пример реализации схемы EAV: “глобальный реестр атрибутов” + “значения атрибутов для конкретного продукта”.

### 1.2 Модалки админ-панели
- **User Edit Modal**: редактирование имени/фамилии, роли (строкой), активности (checkbox).
- **Create Attribute Modal**: создание глобального атрибута (name/dataType/unit).
- **Create Category Modal**: создание категории с parentId (иерархия категорий).

### 1.3 OrderFulfillmentTab (Outbound логистика)
- Подгрузка админского списка заказов (`getAllOrders`) и фильтрация по статусам `Paid/Processing/Shipped/Delivered`.
- Ручная отгрузка: `shipOrder(orderId, carrier, tracking)`.
- Ручная доставка: `deliverOrder(orderId)` (в авто-режиме).
- Авто-логика “симуляции времени”:
  - после 2 “дней” автосоздание отгрузки для Paid
  - после 4 “дней” автоперевод в Delivered для Shipped
- Поллинг каждые 5 секунд для “живого потока”.

Для диплома это можно описывать как “операционный модуль fulfillment” + демонстрация жизненного цикла заказа (Paid → Shipped → Delivered).

## 2) Критичные/важные замечания (что может сломаться или вводить в заблуждение)

### 2.1 Типизация “плывёт” (any в заказах и пользователях)
В `OrderFulfillmentTab`:
```ts
const [orders, setOrders] = useState<any[]>([]);
```
А выше по странице у тебя `AdminOrder` уже описан в `adminApi.ts`. Если заменить `any[]` на `AdminOrder[]`, поймаешь часть багов на этапе компиляции (например, где `delivery`/`shippingAddress` может быть null).

То же касается пользователей: лучше везде держать `User[]`.

### 2.2 Несогласованность типов/ролей: строковая роль в UI vs roleId на сервере
В модалке пользователя роль меняется как строка:
```tsx
setEditingUser({ ...editingUser, role: e.target.value })
```
А сохранение делает маппинг строк → roleId:
```ts
{ 'Admin': 1, 'Customer': 2, 'Employee': 3, 'Supplier': 4 }
```
Риск: если backend вернёт роль иначе (например `admin`/`ADMIN`/`Shipper`), маппинг сломается “тихо”.

Рекомендация: хранить роль как **roleId** в состоянии редактирования, а строку использовать только для отображения (либо получать список ролей с backend).

### 2.3 EAV: dataType есть, но значение всегда строка
Ты создаёшь атрибуты с `dataType = 'string' | 'number' | 'boolean'`, но дальше:
- `SetProductAttributeDTO` использует `value: string`
- инпут значения — всегда текстовый

Это нормально для прототипа, но тогда:
- либо на backend должна быть логика приведения типов и валидации (число/boolean),
- либо на фронте нужно менять ввод в зависимости от `attribute.dataType` (input type=number / checkbox / select).

Иначе атрибут “Weight:number” можно заполнить строкой “abc”, и BI/фильтрация будут ненадёжны.

### 2.4 Fulfillment: симуляция меняет статусы напрямую (минуя shipment-модель)
Сейчас `shipOrder` и `deliverOrder` идут через `/order/{id}/ship` и `/order/{id}/deliver` (adminApi). При этом у тебя уже есть полноценный логистический контур shipments в `logisticsApi.ts`.

Если в бэкенде “отгрузка заказа” должна создавать `Shipment` + `ShipmentStatusHistory`, то прямое обновление статуса заказа может:
- не записать историю,
- не создать tracking сущность,
- не обновить складские движения.

Решение зависит от твоей backend-реализации:
- если `/order/{id}/ship` внутри создаёт Shipment — ок;
- если нет — лучше переводить fulfillment на `logisticsApi.createShipment(...)`, `markShipped`, `markDelivered` (и не дублировать логику).

### 2.5 Поллинг + автособытия могут конфликтовать
У тебя одновременно:
- поллинг `setInterval(silentRefreshOrders, 5000)`
- авто-ship/auto-deliver эффекты, которые тоже вызывают `loadOrders()`

Это может давать лишние запросы и “дёргание” состояния. Не критично, но для качества:
- можно объединить refresh-логику,
- или использовать один источник обновления (например только поллинг, а авто-операции без `loadOrders`, полагаясь на следующий поллинг).

## 1) AuthModal (login/register)

### Что хорошо
- Логин отправляет `sessionId` (`cart_session_id`) вместе с email/password → это логично для серверного **merge анонимной корзины** после авторизации.
- После логина сохраняешь `accessToken` в `sessionStorage` и закрываешь модалку.
- Регистрация показывает экран “Check your email” → соответствует твоему описанию email confirmation.

### Риски/замечания
1) **Дублирование API-слоя**  
   Тут запросы идут прямым `fetch('/api/auth/login')`, а в остальном проекте есть `authApi.ts`/`getHeaders()`. Лучше унифицировать (одна точка входа), иначе при изменении контрактов auth придётся править в нескольких местах.

2) **Маппинг ролей**
   ```ts
   backendRole?.toLowerCase() -> 'admin'|'employee'|'supplier'|'customer'
   ```
   В `UserProfile` у тебя есть ещё `'shipper'`, но в mapping он никогда не возвращается. Если backend реально имеет роль Shipper/Logistics — UI её сейчас “потеряет”.

3) **Refresh token/сессия**
   В модалке сохраняется только access token. Если у тебя в проекте есть refresh-token cookie (как ты описывал), то “продление сессии” должно быть реализовано где-то ещё (иначе после истечения access token всё начнёт отдавать 401). Это не баг этого компонента, но важно проверить в общем фронтенде.

## 2) CartDrawer + CartPage

### Что хорошо
- Чистые компоненты-представления: принимают `items` и callbacks, не держат лишнюю бизнес-логику.
- Корректно считаешь суммы, выводишь количество и subtotal.
- В CartPage есть защита от qty < 1 (кнопка “minus” disabled), удаление отдельной кнопкой.

### Риски/замечания
- В CartPage кнопка корзины с мусоркой сейчас **удаляет товар**, а не уменьшает qty — это ок, но UX может быть неожиданным. (Не критично.)
- `CartItem extends Product` — у тебя `id` строка (из Product), а cartItemId/cartId — числа. Главное, чтобы в логике обновления/удаления ты использовал правильные идентификаторы (в твоём `cartApi` backend ожидает `cartId`/`cartItemId`, а здесь callbacks идут по `id: string`). Нужно проверить связку контейнерного компонента: не теряется ли `cartItemId`.

## 3) CheckoutPage (самое важное)

### Что хорошо
- Пошаговый flow shipping → delivery → payment, есть валидации формы.
- Оплата сделана так, что можно “создать заказ и оплатить позже” (card fields пустые) — это соответствует твоей модели статусов “To Pay”.
- Правильная привязка к `orderApi.checkout(addressData)` и потом `orderApi.payOrder(orderId, ...)`.
- Карточная валидация (min/max, expiry, cvc) сделана аккуратно.

### Важные несостыковки/пробелы
1) **Invoice request в UI есть, но API-вызова нет**  
   Ты собираешь:
   ```ts
   requestInvoice, companyName, taxId, invoiceAddress
   ```
   но после `checkout`/`payOrder` нигде не вызываешь:
   - `POST /order/{id}/request-invoice` (он у тебя есть в `paymentApi.ts`)
   
   То есть пользователь может поставить галку “Request Invoice”, но на backend это не уйдёт. Если инвойс создаётся только по запросу — функционал фактически недоделан.

   Минимальное исправление: после создания заказа (и желательно после успешной оплаты) добавить:
   ```ts
   if (formData.requestInvoice) await requestInvoice(orderId, {...});
   ```

2) **Payment method UI не влияет на видимость card формы**  
   Сейчас блок “Credit/Debit Card” отображается всегда, даже если выбран PayPal/Transfer/Apple Pay. Это может путать и создавать ложное ощущение, что данные карты нужны всегда. Логичнее показывать card форму только при `paymentMethod === 'card'`.

3) **Delivery date нигде не отправляется**
   `deliveryDate` выбирается, но в `addressData`/checkout payload не передаётся. Если backend это не хранит — ок, но тогда это чисто UI-поле. Если хочешь использовать в BI (lead time, ETA vs actual), лучше отправлять и сохранять (хотя бы как EstimatedDeliveryDate).

4) **Смешение терминов Tax ID / EIN**
   В UI Tax ID (EIN), но в Европе мог бы быть VAT ID. Главное — чтобы backend DTO назывался одинаково (ты уже используешь `taxId`).

5) **Ошибка-объект в alert**
   Тут:
   ```ts
   alert('Failed to place order: ' + err)
   ```
   `err` часто будет объектом, получится `[object Object]`. Лучше `String(err?.message ?? err)`.

## 1) OrderModal (My Purchases)

### 1.1 Что реализовано хорошо
- Вкладки **To Pay / To Ship / Shipped / To Review / Returns** с бейджами.
- Оплата “позже” поддержана: заказ в `AwaitingPayment` можно оплатить из вкладки To Pay через `payOrder(...)`.
- Детали заказа (адрес, позиции, итог, трекинг) — всё аккуратно.
- Отзывы можно писать только по Delivered (To Review) — логично.
- Возвраты вынесены в отдельный UI-поток (создание возврата + отмена Pending возврата).
- Есть “скачивание инвойса” через `getInvoice(orderId)` и генерация печатной HTML-страницы.

### 1.2 Важная проблема: return создаётся с неправильным идентификатором
В коде возврата ты используешь `productId` там, где по смыслу нужен **orderItemId**:

```ts
setReturnSelectedItemId(order.items[0].productId)
...
createReturn({
  orderId: selectedOrderForReturn.id,
  orderItemId: returnSelectedItemId, // <-- сюда уходит productId
})
```

И в форме выбора ты тоже используешь `item.productId` как value.

Если бэкенд ожидает `OrderItemId` (что очень похоже на правду, т.к. в ReturnDTO есть поле `orderItemId`), то возвраты будут:
- привязываться неправильно,
- либо вообще падать/создаваться некорректно.

**Правильный вариант**: в DTO заказа (items) должен приходить `orderItemId`, и именно он должен отправляться как `orderItemId`.  
Т.е. `OrderItem` на фронте должен иметь `id` (orderItemId), как минимум.

Сейчас у тебя `OrderItem` из `orderApi.ts` такого поля не содержит — это первопричина.

### 1.3 Несогласованность: фильтрация заказов по `status` строкой
Ты фильтруешь так:
```ts
orders.filter(o => o.status === 'AwaitingPayment')
```

Но параллельно у тебя есть `statusId` и константы статусов. Надёжнее фильтровать по `statusId`, потому что строки могут поменяться (например локализация, PascalCase, пробелы и т.д.).

#### (A) Очень опасно: роль для AdminPage по умолчанию = admin

Ты ранее показывал HomePage, где карточки строятся из `PRODUCTS` (статических), а `addToCart` делает:

Это потенциальный runtime-баг.

Должно быть:
```ts
const handleCategoryFromMenu = (category: string | null) => { ... }
```

Сейчас удаление из корзины:
```ts

```
но **нет вызова** `cartApi.removeCartItem(cartId, cartItemId)`.

То же касается CartDrawer/CartPage: они вызывают `onRemoveItem`, но backend не обновляют.

#### (E) addToCart делает “оптимистичное добавление” с cartId/cartItemId = 0
```ts
cartId: 0,
cartItemId: 0
```
Потом `updateQuantity` пытается “досинхронизироваться” через `fetchCart()`. Это рабочая стратегия, но:
- до синка удаление/обновление могут быть некорректны,
- лучше после `addToCartAPI` сразу вызвать `fetchCart()` и обновить `cartId/cartItemId` корректно.

#### (F) logout endpoint: неправильный путь/регистр
```ts
await fetch('/api/Auth/logout', { method:'POST' })
```
А в других местах у тебя `/api/auth/logout` (нижний регистр). На Linux/в контейнерах это легко даст 404.

Плюс здесь не передаётся `Authorization`/cookie явно — зависит от твоей серверной реализации.

#### (G) authMode не переключается
Ты открываешь AuthModal одинаково и для login и для register:
```ts

onRegisterClick={() => { ... setIsAuthOpen(true); }}
```

но не делаешь `setAuthMode('login'|'register')`. В итоге модалка почти всегда откроется в дефолтном режиме.

## 1) Auth DTO (совпадение + важные нюансы)

### Совпадает
- `AuthenticateRequestDTO` содержит `SessionId` — фронт в `AuthModal` реально отправляет `sessionId` из `sessionStorage`. Это хорошо для merge корзины.

### Нюансы/риски
- `AuthenticateResponseDTO` возвращает `RefreshToken` **в теле ответа**, а ты ранее писал/планировал “refresh token в HttpOnly cookie”.  
  Если в итоге refresh хранится в cookie — лучше **не возвращать refresh token в JSON**, чтобы не увеличивать риски (XSS/утечки). Если пока прототип — ок, но тогда фронту нужно понимать, где брать refresh.

- У тебя есть `RefreshTokenRequestDTO` (AccessToken + RefreshToken).  
  Во фронте **нет** реализации refresh flow (кроме того, что `credentials: 'include'` используется местами). Это значит: после истечения access token всё начнёт падать 401, пока пользователь не перелогинится. Это не “ошибка DTO”, но это важный пункт для следующих шагов.

## 2) Cart DTO (в целом совпадает с фронтом)

### Совпадает

- `UpdateQuantityRequestDTO` использует `CartId`, `ItemId`, `NewQuantity` — фронт в `cartApi.updateCartItem` отправляет `{ cartId, cartItemId, newQuantity }` (по смыслу ок).

### Что проверить по контроллеру (потом, когда пришлёшь)
- В `CartApi.ts` у тебя endpoint `/cart/update` с body `{ cartId, cartItemId, newQuantity }`. DTO на бэке ожидает `ItemId` (а не `CartItemId`). Это нормально, если маппинг сделан, но надо убедиться, что property name совпадает (если не используешь кастомный биндинг).

## 3) Product / Attribute DTO (есть несколько несостыковок с фронтом)

### 3.1 ProductDTO vs фронтовый `Product`
`ProductDTO`:
- `InStock`, `StockLevel` есть — фронт это использует.
- `Active` поля **нет** в `ProductDTO`, но есть `CreateProductDTO.Active` и `UpdateProductDTO.Active`.

Во фронте:
- `CreateProductRequest` использует `isActive?: boolean`
- `Product` интерфейс на фронте **не содержит** `isActive`

И дальше ты в Admin “analytics” пытался делать `products.filter(p => p.isActive)` — это уже было подозрительно.

Рекомендация по контракту:
- либо добавить `bool Active` в `ProductDTO` (чтобы UI мог реально видеть активность товара),
- либо выкинуть `isActive` из фронта и жить только `InStock`/`StockLevel`.

### 3.2 EAV: ProductAttributeDTO не содержит Unit
`ProductAttributeDTO` сейчас:
```csharp
public long Id;
public int ProductId;
public int AttributeId;
public string? AttributeName;
public string? Value;
```
Во фронте `ProductAttributeValue` ожидает ещё `unit?: string`.

Если ты хочешь показывать “Weight: 10 kg”, то в ответе должны быть:
- либо `Unit` прямо в `ProductAttributeDTO`,
- либо фронт должен дополнительно подтягивать AttributeDTO по AttributeId (дорого и неудобно),
- либо API для product attributes должен возвращать объединённую модель.

## 4) Order DTO (важно для возвратов/шипментов)

### 4.1 Самое критичное: в OrderItemDTO нет Id (OrderItemId)
`OrderItemDTO` содержит только:
- `ProductId`, `ProductName`, `Quantity`, `UnitPrice`, `ImageUrl`

Но **логистика и возвраты** у тебя в DTO-слое опираются на `OrderItemId`:
- `CreateShipmentRequestDTO` содержит `ShipmentLineDTO(int OrderItemId, int Quantity)`
- `CreateReturnRequestDTO` содержит `int? OrderItemId`

А во фронте ты уже сделал две логики, где тебе нужен orderItemId:
1) **Returns** (OrderModal): сейчас фронт отправляет `orderItemId = productId` (потому что другого id у него нет).
2) **Shipments** (logistics): создание shipment line также обычно требует orderItemId.

Вывод: чтобы процессы “возврат по позиции заказа” и “shipments по позициям” работали корректно, тебе нужно:
- добавить `public int Id { get; set; }` (или `OrderItemId`) в `OrderItemDTO`,
- и на фронте везде использовать именно его (а `productId` оставить для отображения и связи с каталогом).

Это №1 правка, иначе возвраты/отгрузки будут неточно привязаны.

### 4.2 Shipment в OrderDTO объявлен как non-null
```csharp
public ShipmentInfoDTO Shipment { get; set; } = null!;
```
Во фронте `shipment: ShipmentInfo | null`, и логически shipment часто отсутствует до отгрузки.

Лучше сделать:
```csharp
public ShipmentInfoDTO? Shipment { get; set; }
```
или гарантировать, что API всегда отдаёт объект (например с пустыми полями), но это менее естественно.

### 4.3 ShippingAddressDTO не содержит State/Apartment
`AddressDTO` (для checkout) содержит `State` и `Apartment`.
`ShippingAddressDTO` (в OrderDTO) содержит: `Email, FullName, PhoneNumber, StreetAddress, City, ZipCode, Country` — **без State/Apartment**.

Во фронте:
- admin заказ показывает `state` и часть адресных полей,
- checkout отправляет state/apartment,
- order modal при печати инвойса пытается использовать `addr.state`.

Надо привести к одному: либо добавить `State` и `Apartment` в `ShippingAddressDTO`, либо фронту не ждать их.

### 2.1 Критично: OrderItemDTO не содержит `Id`, но вся логистика/возвраты требуют OrderItemId
Сейчас `OrderItemDTO` (DTO для клиента) не имеет `Id`, но:

### 2.2 Критично: Shipment статусы не согласованы между LogisticsService и OrderService (и фронтом)
У тебя есть минимум 2 “шкалы” статусов shipment:

- `LogisticsService.MarkShippedAsync`:
  ```csharp

  ```
- `LogisticsService.MarkDeliveredAsync`:
  ```csharp
  s.ShipmentStatusId = 3; // Delivered
  ```

- `OrderService.ShipOrderAsync` создаёт shipment так:
  ```csharp

  ```
- `OrderService.DeliverOrderAsync` ставит:
  ```csharp
  shipment.ShipmentStatusId = 4; // Delivered
  ```

На фронте в SupplierPage тоже есть свои ожидания (`shipmentStatusId === 4` delivered и т.п.).

**Вывод:** статусы shipment сейчас конфликтуют (2/3/4 используются по-разному в разных местах). Это гарантированно даст неправильные бейджи/логики UI и неконсистентную аналитику.

**Что сделать:**
- завести единый enum/словарь для ShipmentStatusId (например: 1=Created, 2=Shipped, 3=InTransit, 4=Delivered) и **везде** использовать одинаково.
- убрать “магические числа” и использовать константы/enum.

### 2.3 Критично: двойное списание/двойная логика складских движений (Pay vs Shipment)
Сейчас:

- `OrderService.PayAsync` списывает inventory по строкам заказа (`EventType="ORDER_PAID"`).
- `LogisticsService.CreateShipmentAsync` **ещё раз списывает inventory** (`EventType="WZ"`).
- `OrderService.ShipOrderAsync` не списывает inventory (он только создаёт shipment), но списание уже было в PayAsync.
- `PaymentService.RegisterPaymentAsync` тоже содержит списание inventory (EventType="INVOICE_PAID") — если этот сервис где-то используется параллельно, будет совсем плохо.

**Риск:** при использовании логистического эндпоинта “create shipment” после оплаты, inventory уменьшится дважды.

**Нужно выбрать одну стратегию:**
- либо списание при оплате (и тогда создание shipment не трогает inventory),
- либо списание при отгрузке (а при оплате только резерв/проверка).

Для дипломного прототипа проще оставить “списание при оплате”, а в `LogisticsService.CreateShipmentAsync` убрать deduction (или заменить на проверку/резервирование). И проверить, что `PaymentService` не используется параллельно с `OrderService.PayAsync`.

## 2) Критичные проблемы/ошибки в сущностях

### 2.1 Cart.AddItem создаёт CartItem с `CartId = this.Id` (который часто = 0)
```csharp
_items.Add(new CartItem(this.Id, productId, quantity));
```
Если корзина новая и ещё не сохранена, `this.Id == 0`. При этом в `CartItem`:

**Как исправить:**
- либо в `CartItem` держать `Cart` navigation и при создании делать `new CartItem { ProductId=..., Quantity=..., Cart=this }`
- либо в `AddItem` добавлять `new CartItem { ProductId=..., Quantity=..., CartId = this.Id }` только после того, как Cart уже имеет Id (т.е. cart сначала сохраняется, потом items)
- самый надёжный DDD-стиль: `CartItem` вообще не принимает `cartId` в конструкторе, а связь задаётся EF через navigation.

### 2.2 В Order сущность ShippingAddress не nullable, но не всегда задаётся

Технически ты сейчас используешь `CheckoutAsync` как основной путь (он задаёт address), но `CreateOrderAsync` остаётся “мина”.

**Решение:** либо сделать `OrderShippingAddress?` nullable, либо гарантировать создание адреса всегда.

### 3.1 DataType “number/boolean” на фронте не совпадает с тем, что ждёт `ProductAttribute.SetValue`

### A1) СИДИНГ: ProductAttribute ссылается на несуществующие ProductId (FK-ошибка)
Ты сидишь `Product` только с `Id = 1..3`, но `ProductAttribute` сидится с `ProductId = 4,5,6` и т.д.:

В `AuthService.AuthenticateAsync` уже есть:

```csharp
await _cartService.MergeCartAsync(customer.Id, request.SessionId);
```

### B2) CartController: критичная “security hole” — Update/Remove/Clear без проверки владельца
`GET /api/cart` — ок для анонимной корзины (через X-Session-ID).  
`POST /api/cart/add` — ты сам выставляешь `CustomerId` и `SessionId`, тоже ок.

Но:

- `PUT /api/cart/update`  
- `DELETE /api/cart/{cartId}/item/{cartItemId}`  
- `DELETE /api/cart/{cartId}`

**Правильный подход для прототипа минимум:**
- в `CartService.UpdateQuantityAsync/RemoveFromCartAsync/ClearCartAsync` проверять:
  - либо `cart.CustomerId == currentUserId`, если авторизован
  - либо `cart.SessionId == X-Session-ID`, если анонимный
- и в контроллере прокидывать `customerId/sessionId` в запрос/контекст.

**Нужно:**

### B4) OrderController: `GET /api/order/{id}/invoice` не проверяет владельца заказа

### B5) ProductController: `GET /api/product/{id}` возвращает доменную сущность Product, а не ProductDTO

## 2) DDL базы + AppDbContext: EAV и ограничения подтверждают важные вещи

### 2.1 Constraint на DataType
В SQL:
```sql
CK_Attribute_DataType: DataType in ('json','bool','date','decimal','int','string')
```
Значит фронтенд **не должен** отправлять `dataType="number"` или `dataType="boolean"`. Сейчас в UI создания атрибута у тебя как раз `"number"/"boolean"` встречались.

Итог: при попытке создать атрибут с `number`/`boolean` будет ошибка на уровне БД.

**Правильно:** выровнять фронт под `int|decimal|bool|date|string|json`.

### 2.2 Attribute.Code обязателен и уникален
DDL:
```sql
Code nvarchar(60) NOT NULL
UQ_Attribute_Code UNIQUE
```
А в backend DTO для создания атрибута `Code` нет (и сервис его не задаёт). Даже если EF разрешит пустую строку, смысл каталога атрибутов ломается. Лучше:
- добавить `Code` в `CreateAttributeDTO`,
- валидировать уникальность,
- либо генерировать автоматически из Name.

### 2.3 EAV: CHK_OnlyOneValue
DDL:
```sql
CHK_OnlyOneValue: ровно одно из ValueString/ValueInt/.../ValueSizeID должно быть NOT NULL
```
Это хорошо (контроль качества данных), но тогда важно, чтобы логика `SetValue()` и `UpdateProductAsync()` реально соблюдала типы. Сейчас в `ProductService.UpdateProductAsync` ты ставишь всегда `"string"` — это будет запихивать даже числовые значения в `ValueString`. Формально constraint это пропустит, но аналитика по числам будет хуже.

### 2.4 Важная потенциальная проблема сидинга EF (если он включён)
В твоём `OnModelCreating` ты сидишь **ProductAttribute** для ProductId 4/5/6, но `Product.HasData` у тебя только 1..3. Это при миграции может дать FK-ошибку.
Если в реальной базе эти продукты уже есть — ок, но для “чистой” миграции нужно синхронизировать сидинг.
